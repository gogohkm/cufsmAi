# 코드 검증 프로세스

## 1. 목적

이 문서는 본 프로젝트의 설계 계산 정확도를 실무 적용 수준으로 검증하기 위한 작업 계획서이자 검증 기록서다.

검증의 핵심 질문은 다음 세 가지다.

1. 설계 계산 코드가 AISI S100 기준과 실제로 일치하는가
2. SI 입력값이 내부 US 단위계로 정확히 변환되고, 계산 결과가 다시 SI로 정확히 환산되어 표시되는가
3. 코드 전반에 단위계 혼용 또는 계층 간 단위 해석 불일치가 존재하는가

본 검증 단계에서는 기존 코드를 수정하지 않는다. 발견되는 문제는 이 문서에 위치, 증상, 영향, 근거를 기록한다.

## 2. 검증 원칙

- 런타임 코드 기준으로 전수 추적한다.
- 설계식은 구현 코드, 테스트 코드, 예제 문서, AISI 기준식의 네 축으로 대조한다.
- 단위계는 입력, 브리지, 엔진, 설계 모듈, MCP, UI 표시의 전 구간을 연속 경로로 검증한다.
- 추정이 아닌 증거 기반으로 기록한다.
- 코드 수정은 하지 않고 발견 사항만 누적한다.

## 4. 검증 관점별 작업 분해

### 4.1 AISI 설계식 검증

대상 모듈:

- `python/design/aisi_s100.py`
- `python/design/dsm_strength.py`
- `python/design/global_buckling.py`
- `python/design/shear.py`
- `python/design/interaction.py`
- `python/design/connections.py`
- `python/design/lap_connection.py`
- `python/design/special_topics.py`

검증 항목:

- 조항 참조와 구현 식의 일치 여부
- 계수 `phi`, `Omega`, 한계값, 분기 조건의 정확성
- fallback 계산이 AISI 허용 범위를 벗어나지 않는지
- 입력값 우선순위(`props`, `dsm`, `section`, 자동생성값)가 설계 의도와 맞는지
- 테스트가 실제 설계 위험 구간을 충분히 덮는지

### 4.2 단위 변환 검증

대상 경로:

- WebView 입력
- `webview/js/app.js` 변환 함수
- `src/webview/StcfsdPanel.ts`
- `src/bridge/PythonBridge.ts`
- `python/server.py`
- `python/design/*`
- MCP 경로 `src/mcp/*`

검증 항목:

- SI 입력값이 US 내부값으로 정확히 변환되는지
- ft, in, kip-in, kip-ft, psf, plf, ksi 등 개별 단위의 계수가 정확한지
- 결과값이 표시 계층에서 올바른 단위 타입으로 다시 변환되는지
- formula 문자열, 보고서, 트리, 검증 탭, MCP 응답에서 단위 표기가 일관적인지

### 4.3 단위 혼용 및 계층 불일치 검증

검증 항목:

- 동일 물리량이 서로 다른 단위 타입으로 처리되는 위치
- UI 기본값이 실제 내부 단위와 다른 위치
- 설계용 길이와 해석용 길이의 단위 혼동
- 모멘트 `kip-in`과 `kip-ft`의 혼용
- 압축/휨 DSM 추출 시 해석 응력 기준과 설계 Fy 기준이 섞이는 위치

## 5. 수행 절차

### 단계 1. 코드 인벤토리 작성

- 실행 경로와 설계 계산 진입점을 식별한다.
- 단위 변환 함수와 설계 엔진 호출 경로를 연결한다.

### 단계 2. 단위 흐름 추적표 작성

- 입력 필드별 단위 타입을 식별한다.
- 각 값이 어느 함수에서 US로 바뀌는지 기록한다.
- 결과값이 어느 함수에서 다시 SI로 바뀌는지 기록한다.

### 단계 3. 설계식 코드 검토

- AISI 조항별로 실제 구현 공식을 확인한다.
- 식, 분기 임계값, 보정계수, 경고 로직을 대조한다.

### 단계 4. 테스트 및 수치 검증

- 기존 테스트를 실행한다.
- 예제 기반 수치 재현이 가능한 항목을 재검산한다.
- 테스트 부재 영역은 문서에 공백으로 명시한다.

### 단계 5. 혼용 코드 탐지

- 단위 이름, 변환 함수, 표시식, 보고식, 기본값을 교차 검색한다.
- 같은 변수명이 계층마다 다른 단위를 의미하는지 확인한다.

### 단계 6. 발견 사항 정리

- 오류
- 고위험 의심 사항
- 검증 미흡 영역

## 9. 발견 사항 로그

### F-001

- 상태: `confirmed`
- 분류: `integration`
- 심각도: `high`
- 위치: `python/design/loads/load_combinations.py`, `find_controlling_combo()`
- 내용: 양력 조합 판정을 `d_factor < 1.0`으로 정의하고 있어 실제 양력 지배 조합을 잘못 분류할 수 있다.
- 근거:
  - 168-170행에서 `d_factor = factors.get('D', 0)`, `w_factor = factors.get('W', 0)`, `is_uplift = (d_factor < 1.0 and w_factor != 0)`로 구현되어 있다.
  - 이 조건은 ASD `D + 0.6W`, LRFD `1.2D + 1.0W`처럼 풍하중이 포함되어도 `D` 계수가 1.0 이상이면 양력으로 분류하지 못한다.
  - 실제 지배 여부는 조합명이나 `W` 존재 여부가 아니라, 결과 모멘트/반력의 부호와 양력 해석 목적에 따라 판정해야 한다.
- 영향:
  - `find_controlling_combo()`의 `uplift`가 잘못 비어 있을 수 있다.
  - 후속 `required_strength.py`의 `uplift_R`, 자동 설계 입력, §I6.2.1 검토 경로가 틀어질 수 있다.
- 재현 방법 또는 검증 방법:
  - 풍상향 하중이 지배하는 연속보 입력에서 LRFD `1.2D+1.0W` 조합이 음모멘트 최대를 만들어도 `is_uplift`가 `false`로 남는지 확인한다.

### F-002

- 상태: `confirmed`
- 분류: `AISI-logic`
- 심각도: `critical`
- 위치: `python/design/aisi_s100.py`, `_auto_generate_props()`
- 내용: 자동 생성 경로가 코너 반경을 실제 형상 생성에 반영하지 않고, 압축 응력 상태로 한 번만 수행한 FSM 곡선을 휨 DSM 값 추출에도 재사용한다.
- 근거:
  - 1132-1133행에서 `generate_section(section_type, {'H': H, 'B': B, 'D': D, 't': t})`만 전달하고 `R` 또는 `r`을 전달하지 않는다.
  - 1153-1156행에서 `R`을 `props`에만 사후 기록하므로 실제 단면 형상, 단면성질, DSM 곡선에는 반영되지 않는다.
  - 1167-1178행에서 모든 절점 응력을 `Fy`로 동일 설정한 뒤 `stripmain(...)`을 1회 실행하고, 동일 `result.curve`를 `extract_dsm_values(..., 'P')`와 `extract_dsm_values(..., 'Mxx')`에 동시에 사용한다.
  - 휨 DSM의 `Mcrl`, `Mcrd`는 휨 응력분포 기반 좌굴곡선에서 추출되어야 하므로 압축 응력곡선 재사용은 설계 개념과 맞지 않는다.
- 영향:
  - `props`/`dsm`을 외부에서 주지 않고 `aisi_design`에 직접 들어가는 경로의 자동 계산 결과가 크게 왜곡될 수 있다.
  - 코너 반경 민감 단면, 휨 지배 설계, MCP/자동화 호출 경로에서 잘못된 `Mcrl`, `Mcrd`, `My`가 사용될 수 있다.
- 재현 방법 또는 검증 방법:
  - 동일한 `H/B/D/t`에 서로 다른 `R`을 넣어도 자동 생성 경로 결과가 거의 동일한지 확인한다.
  - 동일 단면에 대해 압축과 휨 하중케이스로 각각 FSM을 수행한 값과 현재 자동 생성된 `Mcrl`, `Mcrd`를 비교한다.
