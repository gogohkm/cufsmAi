작성일: 2026-04-10

## 0. 상태 업데이트

2026-04-10 기준으로 이 문서의 Phase 1~5 구현은 실제 코드에 반영되었다. 현재 상태는 "완전 미구현 계획서"가 아니라 "구현 결과와 잔여 제약을 함께 기록한 기준 문서"다.

이번 구현에서 반영된 범위는 다음과 같다.

- `classify_web_crippling_case()` 기반의 case-driven dispatcher 도입
- `Table G5-1` built-up I-section 지원 추가
- single-web `C/Z`에 대한 `Table G5-2/G5-3` 분기 유지
- `C/Z EOF` overhang bearing에 대한 `Eq. G5-2` 구현
- `Lo > 1.5h` 경계에서는 overhang가 아니라 standard EOF로 fallback하도록 `§G5` 정의를 직접 반영
- hat section `Table G5-4` per-web 강도와 `n_webs` 합산 반영
- multi-web deck `Table G5-5` per-web 강도와 `n_webs` 합산, support fastener spacing `> 18 in.` 시 `unfastened` override 반영
- `C/Z ITF`에 대해 명시 입력 `edge_distance`를 받아 `≥ 1.5h / 2.5h` 요구를 직접 검증
- `§H3` 적용성 메타데이터(`h3_applicable`, `h3_equation_candidate`, `h3_not_applicable_reason`) 반환
- MCP / 설계 UI / 저장·복원 / 결과 패널까지 동일한 입력 모델 연결
- Python 회귀 테스트와 `unittest discover` 수집 경로까지 보완
- 상위 설계 경로에서 `built_up_i`와 `edge_distance`가 실제 `§G5` 계산까지 전달되는 회귀 테스트 추가
- `wc_section_family=auto`일 때 상위 설계 경로가 `section.type`, `section.family_hint`, `section.web_count`를 이용해 hat / built-up I / multi-web family와 `n_webs`를 보수적으로 자동 추론

아직 남아 있는 보수적 제약은 다음과 같다.

- hat section의 `unfastened ETF/ITF` 두 플랜지 행과 built-up I-section의 `unstiffened ETF/ITF` 행은 현재 표 추출 범위가 없어 미구현이다.
- `n_webs > 1`인 `C/Z` 단면을 built-up 또는 nested section 강도로 일반화하지는 않았다.
- `ITF` edge-distance extension requirement는 명시 입력이 있을 때만 검증한다. geometry나 support layout에서 자동 추론하지는 않는다.
- overhang bearing과 hat section에 대한 `§H3` 적용성은 보수적으로 `not applicable` 처리한다.
- 저장/복원 및 `wc_*` 입력 전달 경로에 대한 Node 기반 round-trip 회귀 테스트는 추가했지만, 실제 VS Code WebView 런타임을 띄우는 브라우저/패널 E2E 테스트는 아직 없다.

## 1. 목적

현재 저장소의 web crippling 구현은 single-web `C/Z` 단면에 대한 `Eq. G5-1`과 `Table G5-2/G5-3` 일부 분기까지 반영되어 있다. 이 문서는 이를 AISI S100-16 `§G5`의 전체 적용범위로 확장하기 위한 설계 문서다.

이 문서의 목적은 다음과 같다.

- 구현 범위를 "무엇을 계산할 것인가" 수준이 아니라 "어떤 입력, 어떤 분기, 어떤 파일, 어떤 테스트" 수준으로 명확히 고정한다.
- UI, MCP, Python 설계 엔진이 서로 다른 축약 가정을 가지지 않도록 공통 케이스 모델을 정의한다.
- 실제 코드 수정 전, 단계별 작업 순서와 완료 기준을 문서로 고정한다.

## 2. 현재 상태 요약

현재 구현 상태는 다음과 같다.

- 핵심 계산 함수는 [python/design/shear.py](/abs/path/C:/Coding_Works/cufsmAi/python/design/shear.py#L82) `web_crippling()` 이다.
- 현재 함수는 classifier를 통해 `section_family in {C, Z, hat, multi_web}`, `support in {EOF, IOF, ETF, ITF}`, `fastened in {fastened, unfastened}`, `Lo`, `n_webs`, `bearing_case` 등을 받아 `Eq. G5-1` 또는 `Eq. G5-2`를 선택한다.
- `§H3` 조합식은 [python/design/interaction.py](/abs/path/C:/Coding_Works/cufsmAi/python/design/interaction.py#L52) `combined_bending_web_crippling()` 에서 `single`, `nested_z`, `multi_web` 분기를 지원하고, 상위 설계 경로는 `h3_applicable` 메타데이터를 함께 판단한다.
- 설계 UI는 `wc_fastened`, `wc_web_config`, `wc_section_family`, `wc_flange_condition`, `wc_overhang`, `wc_n_webs`, `wc_support_fastener_spacing`를 받고 저장/복원한다.
- MCP `get_web_crippling`은 현재 구현 범위를 `G5-1` for `C/Z/hat/multi_web`, `G5-2` overhang for `C/Z`로 명시하고 관련 입력을 공개한다.

즉, 현재 구현은 다음을 만족한다.

- single-web `C/Z`
- `Table G5-2/G5-3`
- `Eq. G5-1`
- overhang `Eq. G5-2`
- hat `Table G5-4`
- multi-web `Table G5-5`
- `H3-1/H3-2/H3-3` 조합식 선택
- `h3_applicable` / `h3_not_applicable_reason` 메타데이터
- UI / MCP / Python direct call 입력 정합화

다음은 아직 아니다.

- hat `unfastened ETF/ITF` 행
- built-up / nested `C/Z`의 일반화된 다중 웹 강도
- geometry 기반 `ITF` edge-extension 직접 검증
- overhang / hat case의 `§H3` 직접 적용
- 브라우저/패널 레벨 자동 회귀 테스트

## 3. "full §G5 일반화"의 정의

이 프로젝트에서 `full §G5 일반화`는 아래를 의미한다.

1. 사용자가 케이스를 명시하거나, 현재 단면/배치 정보로부터 `§G5` 적용 case를 일관되게 분류할 수 있다.
2. 분류 결과에 따라 `Eq. G5-1` 또는 `Eq. G5-2` 및 관련 표 계수를 정확히 선택한다.
3. single-web `C/Z`뿐 아니라 overhang, hat, multi-web 계열을 계산할 수 있다.
4. 계산 결과는 수치 `Pn`만이 아니라 "어떤 표/식/가정으로 계산했는지"를 메타데이터로 반환한다.
5. `§H3` 조합검토는 `Pn_wc`를 받아 쓰는 수준을 넘어, 해당 web crippling case가 `H3` 적용 대상인지도 함께 판단한다.
6. UI, MCP, Python API가 같은 case model을 공유한다.

## 4. 비목표

이번 일반화 범위에 포함하지 않는 항목은 아래와 같다.

- 임의 단면의 형상 인식만으로 모든 `§G5` case를 자동 완전 판정하는 기능
- 시각 모델러에서 geometry만 보고 hat/multi-web/built-up 여부를 100% 자동 추론하는 기능
- AISI 외 타 규정의 web crippling 식 추가
- 과거 축약 API를 완전히 제거하는 breaking change

즉, 초기 일반화는 "명시 입력 기반의 정확한 계산"을 우선하고, "자동 추론"은 보조 경로로 둔다.

## 5. 요구사항

### 5.1 입력 요구사항

최소 입력 집합은 다음과 같이 확장되어야 한다.

- `section_family`
  - 후보: `built_up_i`, `C`, `Z`, `hat`, `other_single_web`, `multi_web`
- `support`
  - `EOF`, `IOF`, `ETF`, `ITF`
- `fastened`
  - `fastened`, `unfastened`
- `web_config`
  - `single`, `nested_z`, `multi_web`
- `bearing_case`
  - `end_bearing`, `interior_bearing`, `overhang_bearing`
- `N`
  - bearing length
- `R`
  - inside bend radius
- `h`
  - flat web depth
- `t`
  - thickness
- `theta`
  - bearing angle
- `Lo`
  - overhang length, `Eq. G5-2`용
- `n_webs`
  - multi-web 배수 판정용
- `flange_condition`
  - 최소한 `stiffened_or_partially_stiffened` vs `unstiffened` 구분 필요 여부를 명시적으로 검토

### 5.2 출력 요구사항

반환값은 최소한 아래를 포함해야 한다.

- `Pn`
- `phi`
- `omega`
- `equation`
- `table`
- `section_family`
- `support`
- `fastened`
- `web_config`
- `bearing_case`
- `assumptions`
- `warnings`
- `h3_applicable`
- `h3_equation_candidate`

핵심은 `Pn`만 반환하지 않는 것이다. 나중에 보고서, 검증 대시보드, MCP 응답이 모두 같은 설명을 재사용해야 한다.

## 6. 권장 아키텍처

현재처럼 함수 하나에 분기를 계속 추가하는 방식은 유지보수성이 낮다. 아래 구조로 분리하는 것을 권장한다.

### 6.1 케이스 분류기

새 함수 예시:

```python
classify_web_crippling_case(params) -> dict
```

역할:

- 입력을 정규화한다.
- 어떤 `equation`과 `table`을 써야 하는지 결정한다.
- 누락 입력이 있으면 계산 금지 또는 경고를 반환한다.
- `H3` 적용 후보도 함께 결정한다.

예상 출력 예시:

```python
{
  "equation": "G5-1",
  "table": "G5-2",
  "section_family": "C",
  "support": "ETF",
  "fastened": "fastened",
  "bearing_case": "end_bearing",
  "web_config": "single",
  "h3_equation_candidate": "H3-1",
  "h3_applicable": True,
  "warnings": [],
}
```

### 6.2 식별 계산 함수

권장 분리:

- `web_crippling_g51(case, params)`
- `web_crippling_g52_overhang(case, params)`
- `apply_multi_web_factor(case, result)`

이렇게 나누면 식 자체와 배수/후처리를 분리할 수 있다.

### 6.3 상위 진입점

상위 진입점은 아래 역할만 수행한다.

- case 분류
- case별 계산 함수 호출
- 결과 메타데이터 첨부
- `H3` 적용성 정보 병합

즉, 최종적으로는 아래 형태가 된다.

```python
web_crippling(params) -> classify -> solve -> decorate result
```

목표:

- single-web 전용 모델에서 벗어나 hat, multi-web을 지원

- `section_family='built_up_i'` 추가

### Phase 4. `§H3` 적용성 정교화 [완료]

### 리스크 2. UI가 너무 복잡해질 수 있음

대응:

- 기본은 `simple mode`, 고급은 `advanced G5 inputs` 접이식 영역으로 분리
- 자동 추정 가능 값은 기본 채움

### 리스크 3. H3와 G5의 적용 경계가 사용자에게 모호할 수 있음

대응:

- 결과에 항상 `equation`, `table`, `assumptions`, `h3_applicable` 표시
- "계산 가능"과 "코드가 직접 H3 적용 가능"을 구분해 보여줌

## 11. 완료 판정 기준

아래를 모두 만족하면 `full §G5 일반화`를 완료로 본다.

- `web_crippling()`가 `Eq. G5-1`과 `Eq. G5-2`를 모두 지원
- built-up I, single-web `C/Z`, overhang, hat, multi-web 대표 case를 계산 가능
- 결과에 `equation/table/assumptions/warnings`가 포함됨
- `H3` 적용 가능 여부를 결과에 포함
- MCP와 UI가 같은 입력 모델을 사용
- Python 단위 테스트와 UI/MCP 통합 검증이 추가됨
- [코드검증프로세스.md](/abs/path/C:/Coding_Works/cufsmAi/코드검증프로세스.md)의 `F-038` ~ `F-040` 잔여 범위를 닫을 수 있음

현재 판정:

- 위 항목 중 계산엔진/MCP/UI/단위 테스트 관련 본체는 충족했다.
- 다만 hat `unfastened ETF/ITF`, built-up I의 `unstiffened ETF/ITF`, nested/built-up `C/Z`, geometry 기반 edge-distance 자동 추론, 브라우저/패널 자동 테스트가 남아 있어 "완전 종료"가 아니라 "주요 구현 완료 + 보수 잔여 제약 존재" 상태로 본다.

## 12. 권장 착수 순서

실제 구현은 아래 순서가 가장 안전하다.

1. `shear.py`에 case classifier와 metadata 반환 구조를 먼저 도입한다.
2. 현재 single-web `C/Z` 경로를 classifier 기반으로 바꾼 뒤 기존 테스트를 통과시킨다.
3. overhang `G5-2`를 추가한다.
4. hat/multi-web 배수 규칙을 추가한다.
5. `aisi_s100.py`와 `interaction.py`에서 `H3` 적용성 판단을 연결한다.
6. 마지막에 UI/MCP 입력 확장과 저장/복원을 붙인다.

이 순서를 뒤집으면 UI만 먼저 커지고 계산엔진의 기준 모델이 아직 없는 상태가 되어, 다시 불일치가 생길 가능성이 높다.
